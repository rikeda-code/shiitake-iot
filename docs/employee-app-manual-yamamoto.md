# 山本くん専用「クロードコードで出荷ダッシュボードを作るマニュアル」

対象読者: プログラミング未経験〜初心者
使い方: 1人で最初から順番に読み進める自習形式です。途中に出てくる **🤔 考えてみようボックス** は、答えを埋める前に自分の手を止めて考えるためのスペースです。埋められなくても先に進んでOKです。

このマニュアルは、あなた(山本くん)が作りたい「日々の出荷作業量と人手の関係がわかるダッシュボード」を題材に、最初から最後まで一緒に作っていく形で進みます。

- **作りたいもの**: 日々の出荷作業量と人手の関係がわかるダッシュボード
- **記録したい情報**: その日作成した各商品の数、出荷班の勤務シフト(2点ともスプレッドシート)

  - 分析できるダッシュボード
  - 携帯、パソコンからでもデータの打ち込みが可能
  - デバイスが違うところから入力しても反映される

## 第1章: 全体の流れ

クロードコードでアプリを作る流れは、いつも同じです。

```
① 何を作りたいか言葉にする
      ↓
② 必要な情報(データ)を整理する
      ↓
③ クロードコードにお願いする(プロンプト)
      ↓
④ 出来上がったものを確認する
      ↓
⑤ 直してほしいところを伝える(③と④を繰り返す)
      ↓
⑥ 保存して、みんなが見られるようにする
```

大事なのは、**①〜②を先にやってから③に進むこと**です。ここを飛ばして「なんとなく」お願いすると、欲しいものと違うものができあがりやすくなります。

## 第2章: 事前準備・初期設定

### 2.1 必要なもの

- 会社のメールアドレス
- GitHub アカウント(なければこの章で作ります)
- スマホまたはPC(ブラウザが使えればOK。アプリのインストールは不要)
- 各商品の生産数が入っているGoogleスプレッドシートへのアクセス権
- 出荷班の勤務シフトが入っているGoogleスプレッドシートへのアクセス権

### 2.2 GitHubアカウントを準備する

GitHub(ギットハブ)は、作ったアプリの「置き場所」であり「変更履歴の記録係」です。難しく考えず、**Googleドライブのフォルダに近いもの**とイメージしてください。

1. https://github.com にアクセスし、アカウントを作成する(すでに持っていればログインでOK)
2. 会社のGitHub組織(Organization)に招待されている場合は、届いた招待メールから参加する
3. わからない場合は先輩・管理者に「GitHubの組織に入れてください」と依頼する

### 2.3 リポジトリを用意する

「リポジトリ」とは、1つのアプリ(プロジェクト)を入れる専用フォルダのことです。

- 新しく作る場合: GitHub上で `New repository` から作成し、名前は英数字とハイフンのみ(例: `shukka-dashboard`)にする
- 既存のリポジトリ(例: このリポジトリの `dashboard/` 配下)に追加する場合は、先輩に確認する

### 2.4 Claude Code on the web を開く

このマニュアルでは、ブラウザだけで使える「Claude Code on the web」を使います。パソコンにソフトをインストールする必要はありません。

1. Claude Code on the web にログインする
2. 「新しいセッション」を作成し、2.3で用意したリポジトリを接続する
3. 接続すると、そのリポジトリの中身をクロードコードが読み書きできるようになる

### 2.5 「違うデバイスから入力しても反映される」をゴールから逆算する

ポイントは、**データの保存場所を1か所(共通のGoogleスプレッドシート)に決めておくこと**です。スマホで入力しても、パソコンで入力しても、同じスプレッドシートに書き込まれるようにすれば、どのデバイスで見ても最新の状態が反映されます。アプリ自体がデバイスごとにデータを持つわけではない、とイメージすると理解しやすいです。

```
[スマホ] ─┐
          ├─→ 共通のGoogleスプレッドシート ─→ ダッシュボード表示
[パソコン]─┘
```

そして「携帯、パソコンからでも見られる」を実現する仕組みは、次の1つだけ覚えておけば十分です。

| 方法 | 特徴 |
|---|---|
| **GitHub Pages** で公開する | 作ったアプリ(HTMLファイル)にURLが発行され、誰でもそのURLを開けば見られる。スマホのブラウザでもOK |

具体的な手順は第6章で扱います。

## 第3章: 作る前に考える(設計)

いきなりクロードコードに話しかける前に、紙(かメモアプリ)の上で情報を整理します。この工程を飛ばさないことが、初心者が一番つまずきやすいポイントを避けるコツです。

### 3.1 「何を作るか」を1行で言葉にする

**例:**「出荷数と出荷班の人数の関係が一目でわかるダッシュボード」

### 3.2 データの棚卸し

| 情報 | 入力元 | 入力方法 |
|---|---|---|
| 各商品の生産数 | 出荷班の記録 | Googleスプレッドシートに入力 |
| 出荷班の勤務シフト | シフト表 | Googleスプレッドシートに入力 |

どちらも「すでにスプレッドシートで運用している」データですが、「携帯・パソコンどちらからでも打ち込める」「違うデバイスから入力しても反映される」という要件があるため、**単に読み込むだけでなく、アプリからスプレッドシートへ書き込む仕組み**も必要になります。この点は第6章の対処法とそのまま対応しています。

### 3.3 画面イメージのラフスケッチ

上手・下手は関係ありません。四角と文字だけの落書きで十分です。例えば「上に日別の生産数と人数を重ねたグラフ」「その下に日ごとの入力フォーム」といった並び順のメモがあれば十分です。

## 第4章: プロンプトの打ち込み方のコツ

### 4.1 良いプロンプトの型

以下の順番で書くと、クロードコードは迷わず動けます。

```
【背景】どんな仕事のためのアプリか
【データ】どこに・どんな形式であるデータを使うか(スプレッドシートのURL、列名など)
【欲しいもの】画面に何を表示・入力できてほしいか
【使う場面】誰が、どのデバイスで使うか
【最初の一歩】まず何ができれば十分か(MVP)
```

### 4.2 最初の一発目プロンプトの例

```
出荷作業量と人手の関係を分析するダッシュボードを作りたいです。

【背景】その日の出荷班の人数と、実際に作れた商品数の関係を見て、
人員配置の参考にしたいです。

【データ】
- 各商品の生産数はGoogleスプレッドシートに記録されています(URL: ○○○)
- 出荷班の勤務シフトも別のGoogleスプレッドシートにあります(URL: ○○○)

【欲しいもの】
- 日ごとの「生産数の合計」と「その日の出荷班の人数」を並べたグラフ
- パソコンからもスマホからもデータを見たり入力したりできること
- どのデバイスから入力しても、他のデバイスに反映されること

【最初の一歩】
まずは生産数と出荷班人数を日付ごとに1つのグラフに並べて
表示するだけのシンプルなページを作ってください。
```

このリポジトリの `dashboard/` フォルダには、Googleスプレッドシートと連携した既存のダッシュボードがあります。「◯◯を参考にして」と伝えると、同じ作り方を踏襲してもらいやすくなります。

### 4.3 小さく刻んで頼む

一度に「あれもこれも」と頼むより、MVPができたあとに1つずつ機能を足していく方が、仕上がりの精度が上がります。

- ❌「グラフも入力機能も指標も全部一気に作って」
- ⭕「まずグラフを作って」→ 確認 →「次に入力フォームを足して」→ 確認 →「次に分析指標を足して」

### 4.4 うまく伝わらなかったときの直し方

- 見た目がおかしい → スクリーンショットを撮って「この部分がこう見えてしまう。本当はこうしたい」と伝える
- エラーが出た → エラーメッセージの文章を丸ごとコピーして貼り付ける(要約せずそのまま貼るのがコツ)
- 思っていたものと違う → 「◯◯ではなく△△にしてほしい」と欲しい結果を具体的に言い直す

### 4.5 コードが読めなくても大丈夫

クロードコードはコードそのものを書いてくれる係です。あなたの役目は「何を実現したいか」を伝え、「出来上がったものが期待通りか」を確認することです。

## 第5章: 実際に作る、ステップ・バイ・ステップ

### ステップ0: お手本を見てみる

このリポジトリの `dashboard/` フォルダには、すでにGoogleスプレッドシートと連携したダッシュボードの実例があります。難しく感じたら中身を全部理解する必要はありません。「スプレッドシートと連携したダッシュボードが実際に作れている」というイメージだけ持ち帰ってください。

### ステップ1: MVPを作ってもらう(生産数×人数のグラフ)

第4章のプロンプト例を参考に、クロードコードにMVPをお願いします。ここでは「生産数と出荷班人数を日付ごとに1つのグラフに並べて表示する」ことだけをお願いします。

2つのスプレッドシートを読み込めるようにする設定(共有範囲の変更)が必要になるので、先に第6章6.1を済ませておくとスムーズです。

### ステップ2: アプリから入力できるようにする

「携帯、パソコンからでもデータの打ち込みが可能」という要件は、新しくデータを「書き込む」機能なので、第6章6.2の方式(Google Apps Script)が必要になります。

```
このダッシュボード上から、その日の生産数と出荷班の出勤人数を
入力できるフォームを追加してください。
入力した内容はGoogleスプレッドシートに書き込み、
別のデバイスで開いたときもすぐにグラフへ反映されるようにしてください。
```

### ステップ3: 分析用の指標を追加する

```
「1人あたりの生産数(生産数合計 ÷ 出荷班人数)」を計算して、
グラフの下に指標として表示してください。
```

### ステップ4: 商品ごとの内訳を見られるようにする

```
商品ごとの生産数の内訳も見られるように、
商品を選ぶと生産数の推移が見られるグラフを追加してください。
```

### ステップ5: スマホでの入力・閲覧のしやすさを確認する

パソコンの画面とスマホの画面は見え方が変わります。特に「入力フォーム」はスマホでタップしやすいかどうかが重要です。クロードコードのプレビュー画面をスマホサイズの幅に縮小して確認するか、実際にスマホのブラウザでURLを開いて確認しましょう。

### ステップ6: 保存する(コミット・プッシュ)

クロードコードが作業した内容は、あなたが指示すれば自動的に「保存(コミット)」され、リポジトリに「反映(プッシュ)」されます。「ここまでの変更を保存して」と伝えるだけで大丈夫です。

### ステップ7: みんなに共有する

第6章6.5の手順で公開URLを発行し、そのURLを出荷班のメンバーに共有します。

## 第6章: GitHubや外部アプリを経由するときの対処法

ここは「詰まりやすいポイント」をあらかじめ解消しておく章です。困ったときに戻ってくる場所として使ってください。

### 6.1 Googleスプレッドシート(生産数・シフト)を読み取れるようにする

アプリ側からスプレッドシートを読み込むには、シートを「リンクを知っている人なら閲覧できる」状態に公開しておく必要があります。生産数・シフトの2つのシートどちらにも同じ手順を行います。

1. 対象のGoogleスプレッドシートを開く
2. 右上の「共有」ボタンをクリック
3. 「リンクを取得」で **閲覧者(Viewer)** に設定する
4. 発行されたURLの中の `/d/` と `/edit` の間にある文字列(シートID)をコピーする
5. クロードコードに伝えるときは、このURLまたはシートIDと、シート(タブ)名・列名をあわせて伝える

> ⚠️ 社外秘の情報が含まれるシートを「リンクを知っている人は誰でも閲覧可」にするのは避けてください。個人情報(氏名を含むシフト表など)が入っている場合は、先輩・管理者に公開範囲を必ず確認しましょう。

### 6.2 アプリから入力・書き込みできるようにする(Google Apps Script)

読み取るだけでなく、アプリから新しいデータ(生産数・出勤人数など)を書き込みたい場合は、読み取り専用の公開だけでは足りません。この場合はクロードコードに次のように伝えます。

```
このダッシュボードから入力したデータをGoogleスプレッドシートに書き込めるようにしたいです。
Google Apps Script(GAS)を使ったWebアプリとして、
書き込み用の仕組みを作る方法を教えてください。
```

クロードコードがGoogle Apps Script用のコードを用意してくれるので、その後の「Googleスプレッドシート側でスクリプトを貼り付けて公開する」手順は、クロードコードに1ステップずつ確認しながら進めてください。会社のGoogleアカウントの権限に関わるため、初めての場合は先輩と一緒に行うことをおすすめします。

この仕組みを使うと、「デバイスが違うところから入力しても反映される」という要件も自然に満たせます。すべてのデバイスが同じスプレッドシートに書き込み・読み込みをするためです。

### 6.3 GitHubの基本用語(これだけ覚えればOK)

| 用語 | イメージ |
|---|---|
| リポジトリ | アプリ専用のフォルダ |
| ブランチ | 作業用の下書きスペース。今回はあらかじめ用意された自分専用のブランチを使う |
| コミット | 変更内容の保存 |
| プッシュ | 保存した変更をリポジトリに反映すること |
| プルリクエスト(PR) | 「この変更を本番に取り込んでいいですか?」という確認・レビューの依頼 |
| GitHub Pages | リポジトリの中身をインターネット上のURLとして公開する仕組み |

これらの操作の多くはクロードコードが代わりに行ってくれます。あなたが覚えるべきは「保存して」「公開して」と日本語でお願いすればよい、ということです。

### 6.4 困ったときのチェックリスト

- [ ] 画面に何も表示されない → スプレッドシートの共有設定が「閲覧者」になっているか確認する
- [ ] 「アクセスが拒否されました」と出る → 会社のGoogleアカウントでログインした状態でシートを開き直し、共有範囲を確認する
- [ ] 一部の列だけ反映されない → スプレッドシートの列名(見出し)が、クロードコードに伝えた名前と一致しているか確認する
- [ ] スマホから入力したデータがパソコンに反映されない → 両方とも同じスプレッドシートを見ているか、入力後に画面を更新(リロード)したか確認する
- [ ] それでも解決しない → エラーメッセージ全文をコピーして先輩に共有する

## 第7章: 完成後にやること・用語集・テンプレート集

### 7.1 完成チェックリスト

- [ ] パソコンのブラウザで正しく表示される

- [ ] 別の人が公開URLを開いても、同じデータが見られる

- [ ] 変更内容がコミット・プッシュされている
- [ ] チームにURLを共有した

### 7.2 完成後もアプリは育てられる

一度完成させたあとも、「ここをもっとこうしたい」と気づいたら、いつでもクロードコードに追加のお願いをして育てていくことができます。焦って最初から完璧を目指す必要はありません。

### 7.3 困ったときの相談先

- 操作方法がわからない → このマニュアルの該当章を読み返す
- 会社のスプレッドシート・GitHub組織の権限まわり → 先輩・管理者に相談する
- 個人情報や社外秘データの扱いに迷ったとき → 必ず先輩・管理者に確認してから進める

### 7.4 用語集

| 用語 | 説明 |
|---|---|
| プロンプト | クロードコードへのお願い文・指示文のこと |
| MVP | Minimum Viable Product の略。最初に作る「最低限動くもの」 |
| レスポンシブ対応 | パソコンでもスマホでも見やすいように画面が自動で調整されること |
| リポジトリ | アプリのファイル一式を入れておく専用フォルダ |
| コミット / プッシュ | 変更の保存 / 保存した変更をリポジトリに反映すること |
| GitHub Pages | 作ったアプリをインターネット上のURLとして公開する仕組み |
| Google Apps Script(GAS) | Googleスプレッドシートなどを操作するための仕組み。書き込み機能を作るときに使う |
