## 第1章: 全体の流れ

クロードコードでアプリを作る流れは、実はいつも同じです。

```
① 何を作りたいか言葉にする
      ↓
② 必要な情報(データ)を整理する
      ↓
③ クロードコードにお願いする(プロンプト)
      ↓
④ 出来上がったものを確認する
      ↓
⑤ 直してほしいところを伝える(③と④を繰り返す)
      ↓
⑥ 保存して、みんなが見られるようにする
```

大事なのは、**①〜②を先にやってから③に進むこと**です。ここを飛ばして「なんとなく」お願いすると、欲しいものと違うものができあがりやすくなります。このマニュアルは①〜⑥を順番に体験できるように作られています。

## 第2章: 事前準備・初期設定

### 2.1 必要なもの

- 会社のメールアドレス
- GitHub アカウント(なければこの章で作ります)
- スマホまたはPC(ブラウザが使えればOK。アプリのインストールは不要)
- 使いたいデータの場所(Googleスプレッドシートなど)へのアクセス権

### 2.2 GitHubアカウントを準備する

GitHub(ギットハブ)は、作ったアプリの「置き場所」であり「変更履歴の記録係」です。難しく考えず、**Googleドライブのフォルダに近いもの**とイメージしてください。

1. https://github.com にアクセスし、アカウントを作成する(すでに持っていればログインでOK)
2. 会社のGitHub組織(Organization)に招待されている場合は、届いた招待メールから参加する
3. わからない場合は先輩・管理者に「GitHubの組織に入れてください」と依頼する

### 2.3 リポジトリを用意する

「リポジトリ」とは、1つのアプリ(プロジェクト)を入れる専用フォルダのことです。

- 新しく作る場合: GitHub上で `New repository` から作成し、名前は英数字とハイフンのみ(例: `saibai-nisshi`, `shukka-dashboard`)にする
- 既存のリポジトリに追加する場合: 先輩に「どのリポジトリに追加すればいいか」を確認する

### 2.5 スマホ・他の人からも見られる、をゴールから逆算する

青木くんも山本くんも「他のデバイスから見られる」「携帯からも使える」が要件です。これを実現する仕組みは主に2つあります。

## 第3章: 作る前に考える(設計)

いきなりクロードコードに話しかける前に、紙(かメモアプリ)の上で3つを整理します。この工程を飛ばさないことが、初心者が一番つまずきやすいポイントを避けるコツです。

### 3.1 「何を作るか」を1行で言葉にする

例:
- 青木くん:「毎日の椎茸コンテナの様子を写真とコメントで記録して、誰でも見返せる日誌」
- 山本くん:「出荷数と出荷班の人数の関係が一目でわかるダッシュボード」

### 3.2 データの棚卸し表を作る

「どんな情報を」「どこから」「どうやって」入れるのかを表にします。

**青木くんの例:**

| 情報 | 入力元 | 入力方法 |
|---|---|---|
| コンテナの写真 | スマホのカメラ | 毎日その場で撮影しアップロード |
| コメント | 本人の頭の中 | テキスト入力 |
| 収穫量・菌床重量 | 実測 | スプレッドシートに記録済み |
| 温度 | 別の温度計測アプリ | そのアプリからエクスポート/連携 |

**山本くんの例:**

| 情報 | 入力元 | 入力方法 |
|---|---|---|
| 各商品の生産数 | 出荷班の記録 | スプレッドシート入力 |
| 出荷班の勤務シフト | シフト表 | スプレッドシート入力 |

### 3.3 画面イメージのラフスケッチ

上手・下手は関係ありません。四角と文字だけの落書きで十分です。「上に写真、下にコメント」「グラフが上、表が下」くらいのメモがあれば、クロードコードへの指示が驚くほど具体的になります。

### 3.4 最初のゴール(MVP)を決める

最初から全部の機能を一度に作ろうとしないことが、遠回りに見えて一番の近道です。「まず動くもの」を最初に作り、そこから育てていきます。

- 青木くんのMVP例:「スプレッドシートの収穫量・菌床重量を表で表示するだけ」
- 山本くんのMVP例:「生産数とシフト人数を1つのグラフに並べて表示するだけ」

## 第4章: プロンプトの打ち込み方のコツ

### 4.1 良いプロンプトの型

以下の順番で書くと、クロードコードは迷わず動けます。

```
【背景】どんな仕事のためのアプリか
【データ】どこに・どんな形式であるデータを使うか(スプレッドシートのURL、列名など)
【欲しいもの】画面に何を表示・入力できてほしいか
【使う場面】誰が、どのデバイスで使うか
【最初の一歩】まず何ができれば十分か(MVP)
```

### 4.2 最初の一発目プロンプトの例

**青木くん版:**

```
椎茸の栽培日誌アプリを作りたいです。

【背景】自分が担当している椎茸コンテナの状態を毎日記録し、
あとで振り返れるようにしたいです。他の担当者や上長も見られるようにしたいです。

【データ】
- 収穫量と菌床重量はGoogleスプレッドシートに記録されています(URL: ○○○)
- その日のコンテナの写真とコメントは、その都度入力したいです
- 温度データは別アプリからエクスポートしたCSVを使う予定です

【欲しいもの】
- 日付ごとに「写真・コメント・収穫量・菌床重量・温度」が一覧できる画面
- 自分の担当コンテナだけを絞り込んで見られる機能
- スマホのブラウザで見やすいレイアウト

【最初の一歩】
まずはスプレッドシートの収穫量と菌床重量を、日付順の一覧表として
表示するだけのシンプルなページを作ってください。
```

**山本くん版:**

```
出荷作業量と人手の関係を分析するダッシュボードを作りたいです。

【背景】その日の出荷班の人数と、実際に作れた商品数の関係を見て、
人員配置の参考にしたいです。

【データ】
- 各商品の生産数はGoogleスプレッドシートに記録されています(URL: ○○○)
- 出荷班の勤務シフトも別のGoogleスプレッドシートにあります(URL: ○○○)

【欲しいもの】
- 日ごとの「生産数の合計」と「その日の出荷班の人数」を並べたグラフ
- パソコンからもスマホからもデータを見たり入力したりできること
- どのデバイスから入力しても、他のデバイスに反映されること

【最初の一歩】
まずは生産数と出荷班人数を日付ごとに1つのグラフに並べて
表示するだけのシンプルなページを作ってください。
```

### 4.3 小さく刻んで頼む

一度に「あれもこれも」と頼むより、MVPができたあとに1つずつ機能を足していく方が、仕上がりの精度が上がります。

- ❌「日誌も写真もグラフも温度連携も全部一気に作って」
- ⭕「まず一覧表を作って」→ 確認 →「次に写真アップロード機能を足して」→ 確認 →「次にスマホ表示を整えて」

### 4.4 うまく伝わらなかったときの直し方

- 見た目がおかしい → スクリーンショットを撮って「この部分がこう見えてしまう。本当はこうしたい」と伝える
- エラーが出た → エラーメッセージの文章を丸ごとコピーして貼り付ける(要約せずそのまま貼るのがコツ)
- 思っていたものと違う → 「なぜそう作ったと思う?」と聞かずに、「◯◯ではなく△△にしてほしい」と欲しい結果を具体的に言い直す

### 4.5 コードが読めなくても大丈夫

クロードコードはコードそのものを書いてくれる係です。あなたの役目は「何を実現したいか」を伝え、「出来上がったものが期待通りか」を確認することです。コードの中身を全部理解する必要はありません。ただし、**動きを日本語で説明できるくらいには目を通しておく**と、後で機能を追加するお願いがスムーズになります。

> #### 🤔 考えてみようボックス

>
> ```
> 【背景】
> 【データ】
> 【欲しいもの】
> 【使う場面】
> 【最初の一歩】
> ```

## 第5章: 実際に作る、ステップ・バイ・ステップ

### ステップ0: お手本を見てみる

このリポジトリには、すでに1つお手本があります。`dashboard/index.html` という1枚のファイルで、以下のことをしています。

- Googleスプレッドシートの内容を、プログラムから直接読み取っている
- 読み取ったデータを月ごとに集計している
- グラフ(棒グラフ)として画面に表示している
- スマホでも見やすいようにレイアウトが調整されている(レスポンシブ対応)

難しく感じたら大丈夫です。中身を全部理解する必要はありません。「1つのファイルだけで、スプレッドシートと連携したアプリが作れる」というイメージだけ持ち帰ってください。青木くん・山本くんのアプリも、基本はこの延長線上で作れます。

### ステップ1: 土台を作ってもらう

第4章で書いたプロンプトをクロードコードに送ります。ここではMVP(最初のゴール)だけをお願いします。

### ステップ2: データ元をつなぐ

スプレッドシートを読み込めるようにする設定が必要です。手順は第6章にまとめてあります。先に該当箇所だけ済ませてからステップ1に戻っても構いません。

### ステップ3: 見た目・機能を1つずつ育てる

MVPができたら、確認しながら機能を1つずつ追加していきます。

- 「写真をアップロードできるようにして」
- 「自分の担当コンテナだけで絞り込めるようにして」
- 「グラフの色をもう少し見やすくして」

### ステップ4: スマホでの見え方を確認する

パソコンの画面とスマホの画面は見え方が変わります。クロードコードのプレビュー画面をスマホサイズの幅に縮小して確認するか、実際にスマホのブラウザでURLを開いて確認しましょう。

「スマホで見たときにボタンが小さすぎる」「文字が画面からはみ出す」など気づいた点はそのままクロードコードに伝えればOKです。

### ステップ5: 保存する(コミット・プッシュ)

クロードコードが作業した内容は、あなたが指示すれば自動的に「保存(コミット)」され、リポジトリに「反映(プッシュ)」されます。難しい操作を自分で覚える必要はなく、「ここまでの変更を保存して」と伝えるだけで大丈夫です。

- **コミット**: 変更内容に「これはこういう変更です」というメモを付けて記録すること
- **プッシュ**: 記録した変更をリポジトリ(みんなが見る場所)に送ること

### ステップ6: みんなに共有する

第6章の手順で公開URLを発行し、そのURLをチームに共有します。

## 第6章: GitHubや外部アプリを経由するときの対処法

ここは「詰まりやすいポイント」をあらかじめ解消しておく章です。困ったときに戻ってくる場所として使ってください。

### 6.1 Googleスプレッドシートを読み取れるようにする

アプリ側からスプレッドシートを読み込むには、シートを「リンクを知っている人なら閲覧できる」状態に公開しておく必要があります(お手本の`dashboard/index.html`もこの方式です)。

1. 対象のGoogleスプレッドシートを開く
2. 右上の「共有」ボタンをクリック
3. 「リンクを取得」で **閲覧者(Viewer)** に設定する
4. 発行されたURLの中の `/d/` と `/edit` の間にある文字列(シートID)をコピーする
5. クロードコードに「このスプレッドシートを使いたい」と伝えるときは、このURLまたはシートIDと、シート(タブ)名・列名をあわせて伝える

> ⚠️ 社外秘の情報が含まれるシートを「リンクを知っている人は誰でも閲覧可」にするのは避けてください。個人情報や取引先情報が入っている場合は、先輩・管理者に公開範囲を必ず確認しましょう。

### 6.2 スプレッドシートに書き込みたい場合(入力フォームなど)

読み取るだけでなく、アプリから新しいデータを書き込みたい場合(例: コメント入力、写真アップロードの記録)は、読み取り専用の公開だけでは足りません。この場合はクロードコードに次のように伝えます。

```
このアプリから入力したデータをGoogleスプレッドシートに書き込めるようにしたいです。
Google Apps Script(GAS)を使ったWebアプリとして、
書き込み用の仕組みを作る方法を教えてください。
```

クロードコードがGoogle Apps Script用のコードを用意してくれるので、その後の「Googleスプレッドシート側でスクリプトを貼り付けて公開する」手順は、クロードコードに1ステップずつ確認しながら進めてください。この部分は会社のGoogleアカウントの権限に関わるため、初めての場合は先輩と一緒に行うことをおすすめします。

### 6.3 写真をアップロードする機能

写真データはスプレッドシートには不向きです。一般的には次のような構成になります。

- 写真本体 → Googleドライブなどの保存先にアップロード
- 「いつ・誰が・どのコンテナの写真か」という情報だけ → スプレッドシートに記録

クロードコードに「写真はGoogleドライブに保存し、スプレッドシートには写真へのリンクだけを記録したい」と伝えると、その構成で組んでもらえます。

### 6.4 温度データなど、別アプリのデータを取り込みたい場合

別アプリ(温度計測アプリなど)からデータを取り込む方法は主に3つあります。会社の状況に合わせて、まずは一番簡単な方法から試しましょう。

| 方法 | 難易度 | 説明 |
|---|---|---|
| CSVエクスポート→手動アップロード | 簡単 | 温度アプリから書き出したファイルを、そのつどアプリに読み込ませる |
| 温度アプリ→スプレッドシートに自動連携 | 中 | 温度アプリ側の連携機能(あれば)を使い、スプレッドシートに自動で貯める |
| API連携 | やや難しい | 温度アプリが公開しているAPIを使い、直接データを取りに行く。会社のIT担当・提供元への確認が必要な場合が多い |

まずは自分たちだけで完結する「CSVエクスポート」から始め、慣れてきたら自動連携を検討する、という順番がおすすめです。判断に迷ったら先輩や情報システム担当に相談してください。

### 6.5 GitHubの基本用語(これだけ覚えればOK)

| 用語 | イメージ |
|---|---|
| リポジトリ | アプリ専用のフォルダ |
| ブランチ | 作業用の下書きスペース。今回はあらかじめ用意された自分専用のブランチを使う |
| コミット | 変更内容の保存 |
| プッシュ | 保存した変更をリポジトリに反映すること |
| プルリクエスト(PR) | 「この変更を本番に取り込んでいいですか?」という確認・レビューの依頼 |
| GitHub Pages | リポジトリの中身をインターネット上のURLとして公開する仕組み |

これらの操作の多くはクロードコードが代わりに行ってくれます。あなたが覚えるべきは「保存して」「公開して」と日本語でお願いすればよい、ということです。

## 第8章: ケーススタディ2: 山本くんの出荷ダッシュボードを作る

### 8.1 要件を確認する

- 作りたいもの: 日々の出荷作業量と人手の関係がわかるダッシュボード
- 記録する情報: その日作成した各商品の数、出荷班の勤務シフト(いずれもスプレッドシート)
- 必要な機能:
  - 分析できるダッシュボード
  - 携帯・パソコンどちらからもデータ入力が可能
  - 違うデバイスから入力しても反映される

### 8.2 「違うデバイスから入力しても反映される」の考え方

ポイントは、**データの保存場所を1か所(共通のスプレッドシート)に決めておくこと**です。スマホで入力しても、パソコンで入力しても、同じスプレッドシートに書き込まれるようにすれば、どのデバイスで見ても最新の状態が反映されます。アプリ自体がデバイスごとにデータを持つわけではない、とイメージすると理解しやすいです。

```
[スマホ] ─┐
          ├─→ 共通のGoogleスプレッドシート ─→ ダッシュボード表示
[パソコン]─┘
```

### 8.3 MVP → 育てる順番の例

1. **MVP**: 生産数と出荷班人数を、日付ごとに1つのグラフに並べて表示する
2. 生産数・シフト人数をアプリ側からも入力できるようにする(6.2の方式)
3. 「1人あたりの生産数」など、分析用の指標を追加する
4. 商品ごとの内訳が見られるようにする
5. スマホでの入力・閲覧の使いやすさを調整する
6. GitHub Pagesで公開し、URLをチームに共有する

### 8.4 各段階でのプロンプト例

**手順1のお願い方の例(既存のdashboard/index.htmlを参考にする):**
```
dashboard/index.html を参考にして、新しいダッシュボードを作りたいです。
生産数のスプレッドシートとシフト人数のスプレッドシートの2つを読み込み、
日付を横軸にして「生産数の合計」と「出荷班人数」を1つのグラフに
重ねて表示してください。
```

**手順3のお願い方の例:**
```
「1人あたりの生産数(生産数合計 ÷ 出荷班人数)」を計算して、
グラフの下に指標として表示してください。
```

### 9.2 完成後もアプリは育てられる

一度完成させたあとも、「ここをもっとこうしたい」と気づいたら、いつでもクロードコードに追加のお願いをして育てていくことができます。焦って最初から完璧を目指す必要はありません。

### 9.3 困ったときの相談先

- 操作方法がわからない → このマニュアルの該当章を読み返す
- 会社のスプレッドシート・GitHub組織の権限まわり → 先輩・管理者に相談する
- 個人情報や社外秘データの扱いに迷ったとき → 必ず先輩・管理者に確認してから進める

### 9.4 用語集

| 用語 | 説明 |
|---|---|
| プロンプト | クロードコードへのお願い文・指示文のこと |
| MVP | Minimum Viable Product の略。最初に作る「最低限動くもの」 |
| レスポンシブ対応 | パソコンでもスマホでも見やすいように画面が自動で調整されること |
| リポジトリ | アプリのファイル一式を入れておく専用フォルダ |
| コミット / プッシュ | 変更の保存 / 保存した変更をリポジトリに反映すること |
| GitHub Pages | 作ったアプリをインターネット上のURLとして公開する仕組み |
| Google Apps Script(GAS) | Googleスプレッドシートなどを操作するための仕組み。書き込み機能を作るときに使う |
