## 第1章: 全体の流れ

クロードコードでアプリを作る流れは、いつも同じです。

```
① 何を作りたいか言葉にする
      ↓
② 必要な情報(データ)を整理する
      ↓
③ クロードコードにお願いする(プロンプト)
      ↓
④ 出来上がったものを確認する
      ↓
⑤ 直してほしいところを伝える(③と④を繰り返す)
      ↓
⑥ 保存して、みんなが見られるようにする
```

大事なのは、**①〜②を先にやってから③に進むこと**です。ここを飛ばして「なんとなく」お願いすると、欲しいものと違うものができあがりやすくなります。

## 第2章: 事前準備・初期設定

### 2.1 必要なもの

- 会社のメールアドレス
- GitHub アカウント(なければこの章で作ります)
- スマホまたはPC(ブラウザが使えればOK。アプリのインストールは不要)
- 収穫量・菌床重量が入っているGoogleスプレッドシートへのアクセス権
- 温度を計測しているアプリへのアクセス

### 2.2 GitHubアカウントを準備する

GitHub(ギットハブ)は、作ったアプリの「置き場所」であり「変更履歴の記録係」です。難しく考えず、**Googleドライブのフォルダに近いもの**とイメージしてください。

1. https://github.com にアクセスし、アカウントを作成する(すでに持っていればログインでOK)
2. 会社のGitHub組織(Organization)に招待されている場合は、届いた招待メールから参加する
3. わからない場合は先輩・管理者に「GitHubの組織に入れてください」と依頼する

### 2.3 リポジトリを用意する

「リポジトリ」とは、1つのアプリ(プロジェクト)を入れる専用フォルダのことです。

- 新しく作る場合: GitHub上で `New repository` から作成し、名前は英数字とハイフンのみ(例: `saibai-nisshi`)にする
- 既存のリポジトリ(例: このリポジトリの `dashboard/` 配下)に追加する場合は、先輩に確認する

### 2.4 Claude Code on the web を開く

このマニュアルでは、ブラウザだけで使える「Claude Code on the web」を使います。パソコンにソフトをインストールする必要はありません。

1. Claude Code on the web にログインする
2. 「新しいセッション」を作成し、2.3で用意したリポジトリを接続する
3. 接続すると、そのリポジトリの中身をクロードコードが読み書きできるようになる

### 2.5 「他のデバイスから見られる」をゴールから逆算する

「他の人も他のデバイスから見られる」「携帯からでもスムーズに使える」を実現する仕組みは、次の1つだけ覚えておけば十分です。

| 方法 | 特徴 |
|---|---|
| **GitHub Pages** で公開する | 作ったアプリ(HTMLファイル)にURLが発行され、誰でもそのURLを開けば見られる。スマホのブラウザでもOK |

つまり「公開する」= 難しい設定ではなく、**1つのURLをみんなに配る**ということです。具体的な手順は第6章で扱います。

## 第3章: 作る前に考える(設計)

いきなりクロードコードに話しかける前に、紙(かメモアプリ)の上で情報を整理します。この工程を飛ばさないことが、初心者が一番つまずきやすいポイントを避けるコツです。

### 3.1 「何を作るか」を1行で言葉にする

**例:**「毎日の椎茸コンテナの様子を写真とコメントで記録して、誰でも見返せる日誌」

### 3.2 データの棚卸し

| 情報 | 入力元 | 入力方法 |
|---|---|---|
| コンテナの写真 | スマホのカメラ | 毎日その場で撮影しアップロード |
| コメント | 本人の頭の中 | テキスト入力 |
| 収穫量・菌床重量 | 実測 | すでにスプレッドシートに記録済み |
| 温度 | 別の温度計測アプリ | そのアプリからエクスポート/連携 |

このうち「収穫量・菌床重量」は**すでにあるデータを読み込むだけ**、「写真・コメント」は**新しく記録する仕組みを作る**、「温度」は**別アプリとの連携方法を考える**、という3種類の性質が混ざっている点がポイントです。この違いは第6章の対処法とそのまま対応しています。

### 3.3 画面イメージのラフスケッチ

上手・下手は関係ありません。四角と文字だけの落書きで十分です。例えば「上に日付とコンテナ選択」「その下に写真」「さらに下にコメント・収穫量・温度」といった並び順のメモがあれば十分です。

最初から全部の機能を一度に作ろうとしないことが、遠回りに見えて一番の近道です。

**MVP例:** スプレッドシートの収穫量・菌床重量を、日付順の一覧表として表示するだけ

## 第4章: プロンプトの打ち込み方のコツ

### 4.1 良いプロンプトの型

以下の順番で書くと、クロードコードは迷わず動けます。

```
【背景】どんな仕事のためのアプリか
【データ】どこに・どんな形式であるデータを使うか(スプレッドシートのURL、列名など)
【欲しいもの】画面に何を表示・入力できてほしいか
【使う場面】誰が、どのデバイスで使うか
【最初の一歩】まず何ができれば十分か(MVP)
```

### 4.2 最初の一発目プロンプトの例

```
椎茸の栽培日誌アプリを作りたいです。

【背景】自分が担当している椎茸コンテナの状態を毎日記録し、
あとで振り返れるようにしたいです。他の担当者や上長も見られるようにしたいです。

【データ】
- 収穫量と菌床重量はGoogleスプレッドシートに記録されています(URL: ○○○)
- その日のコンテナの写真とコメントは、その都度入力したいです
- 温度データは別アプリからエクスポートしたCSVを使う予定です

【欲しいもの】
- 日付ごとに「写真・コメント・収穫量・菌床重量・温度」が一覧できる画面
- 自分の担当コンテナだけを絞り込んで見られる機能
- スマホのブラウザで見やすいレイアウト

【最初の一歩】
まずはスプレッドシートの収穫量と菌床重量を、日付順の一覧表として
表示するだけのシンプルなページを作ってください。
```

### 4.3 小さく刻んで頼む

一度に「あれもこれも」と頼むより、MVPができたあとに1つずつ機能を足していく方が、仕上がりの精度が上がります。

- ❌「日誌も写真もグラフも温度連携も全部一気に作って」
- ⭕「まず一覧表を作って」→ 確認 →「次にコンテナ絞り込みを足して」→ 確認 →「次に写真アップロードを足して」

### 4.4 うまく伝わらなかったときの直し方

- 見た目がおかしい → スクリーンショットを撮って「この部分がこう見えてしまう。本当はこうしたい」と伝える
- エラーが出た → エラーメッセージの文章を丸ごとコピーして貼り付ける(要約せずそのまま貼るのがコツ)
- 思っていたものと違う → 「◯◯ではなく△△にしてほしい」と欲しい結果を具体的に言い直す

### 4.5 コードが読めなくても大丈夫

クロードコードはコードそのものを書いてくれる係です。あなたの役目は「何を実現したいか」を伝え、「出来上がったものが期待通りか」を確認することです。

> 【最初の一歩】

## 第5章: 実際に作る、ステップ・バイ・ステップ

### ステップ0: お手本を見てみる

このリポジトリの `dashboard/` フォルダには、すでにGoogleスプレッドシートと連携したダッシュボードの実例があります。難しく感じたら中身を全部理解する必要はありません。「スプレッドシートと連携したアプリが実際に作れている」というイメージだけ持ち帰ってください。

### ステップ1: MVPを作ってもらう(一覧表)

第4章のプロンプト例を参考に、クロードコードにMVPをお願いします。ここでは「スプレッドシートの収穫量・菌床重量を日付順の一覧表として表示する」ことだけをお願いします。

スプレッドシートを読み込めるようにする設定(共有範囲の変更)が必要になるので、先に第6章6.1を済ませておくとスムーズです。

### ステップ2: 自分の担当コンテナで絞り込めるようにする

```
一覧表に「コンテナ名」で絞り込めるプルダウンを追加してください。
選んだコンテナのデータだけが表示されるようにしてください。
```

### ステップ3: コメント入力機能を追加する

コメントは新しくデータを「書き込む」機能なので、第6章6.2の方式(Google Apps Script)が必要になります。

```
日付・コンテナごとにコメントを入力できる欄を追加したいです。
入力したコメントはGoogleスプレッドシートに保存し、
次に開いたときも表示されるようにしてください。
```

### ステップ4: 写真アップロード機能を追加する

写真は第6章6.3の方式(Googleドライブ + リンクだけスプレッドシートに記録)を使います。

```
その日のコンテナの写真をアップロードできるようにしたいです。
写真本体はGoogleドライブに保存し、スプレッドシートには
写真へのリンクと日付・コンテナ名だけを記録するようにしてください。
アップロードした写真は一覧画面のその日の行に表示してください。
```

### ステップ5: 温度データを取り込む

まずは一番簡単な方法から試します(詳しくは第6章6.4)。

```
別の温度計測アプリからエクスポートしたCSVファイルを取り込んで、
日付・コンテナごとの温度を一覧表に追加したいです。
CSVをアップロードすると反映される形にしてください。
```

### ステップ6: スマホでの見え方を確認する

パソコンの画面とスマホの画面は見え方が変わります。クロードコードのプレビュー画面をスマホサイズの幅に縮小して確認するか、実際にスマホのブラウザでURLを開いて確認しましょう。「ボタンが小さすぎる」「写真がはみ出す」など気づいた点はそのままクロードコードに伝えればOKです。

### ステップ7: 保存する(コミット・プッシュ)

クロードコードが作業した内容は、あなたが指示すれば自動的に「保存(コミット)」され、リポジトリに「反映(プッシュ)」されます。「ここまでの変更を保存して」と伝えるだけで大丈夫です。

### ステップ8: みんなに共有する

第6章6.5の手順で公開URLを発行し、そのURLをチームに共有します。

## 第6章: GitHubや外部アプリを経由するときの対処法

ここは「詰まりやすいポイント」をあらかじめ解消しておく章です。困ったときに戻ってくる場所として使ってください。

### 6.1 Googleスプレッドシート(収穫量・菌床重量)を読み取れるようにする

アプリ側からスプレッドシートを読み込むには、シートを「リンクを知っている人なら閲覧できる」状態に公開しておく必要があります。

1. 対象のGoogleスプレッドシートを開く
2. 右上の「共有」ボタンをクリック
3. 「リンクを取得」で **閲覧者(Viewer)** に設定する
4. 発行されたURLの中の `/d/` と `/edit` の間にある文字列(シートID)をコピーする
5. クロードコードに伝えるときは、このURLまたはシートIDと、シート(タブ)名・列名をあわせて伝える

> ⚠️ 社外秘の情報が含まれるシートを「リンクを知っている人は誰でも閲覧可」にするのは避けてください。個人情報や取引先情報が入っている場合は、先輩・管理者に公開範囲を必ず確認しましょう。

### 6.2 コメントを書き込めるようにする(Google Apps Script)

読み取るだけでなく、アプリから新しいデータ(コメントなど)を書き込みたい場合は、読み取り専用の公開だけでは足りません。この場合はクロードコードに次のように伝えます。

```
このアプリから入力したコメントをGoogleスプレッドシートに書き込めるようにしたいです。
Google Apps Script(GAS)を使ったWebアプリとして、
書き込み用の仕組みを作る方法を教えてください。
```

クロードコードがGoogle Apps Script用のコードを用意してくれるので、その後の「Googleスプレッドシート側でスクリプトを貼り付けて公開する」手順は、クロードコードに1ステップずつ確認しながら進めてください。会社のGoogleアカウントの権限に関わるため、初めての場合は先輩と一緒に行うことをおすすめします。

### 6.3 写真をアップロードする機能

写真データはスプレッドシートには不向きです。一般的には次のような構成になります。

- 写真本体 → Googleドライブなどの保存先にアップロード
- 「いつ・どのコンテナの写真か」という情報だけ → スプレッドシートに記録

クロードコードに「写真はGoogleドライブに保存し、スプレッドシートには写真へのリンクだけを記録したい」と伝えると、その構成で組んでもらえます。

### 6.4 温度データなど、別アプリのデータを取り込みたい場合

| 方法 | 難易度 | 説明 |
|---|---|---|
| CSVエクスポート→手動アップロード | 簡単 | 温度アプリから書き出したファイルを、そのつどアプリに読み込ませる |
| 温度アプリ→スプレッドシートに自動連携 | 中 | 温度アプリ側の連携機能(あれば)を使い、スプレッドシートに自動で貯める |
| API連携 | やや難しい | 温度アプリが公開しているAPIを使い、直接データを取りに行く。会社のIT担当・提供元への確認が必要な場合が多い |

まずは自分たちだけで完結する「CSVエクスポート」から始め、慣れてきたら自動連携を検討する、という順番がおすすめです。判断に迷ったら先輩や情報システム担当に相談してください。

### 6.5 GitHubの基本用語(これだけ覚えればOK)

| 用語 | イメージ |
|---|---|
| リポジトリ | アプリ専用のフォルダ |
| ブランチ | 作業用の下書きスペース。今回はあらかじめ用意された自分専用のブランチを使う |
| コミット | 変更内容の保存 |
| プッシュ | 保存した変更をリポジトリに反映すること |
| プルリクエスト(PR) | 「この変更を本番に取り込んでいいですか?」という確認・レビューの依頼 |
| GitHub Pages | リポジトリの中身をインターネット上のURLとして公開する仕組み |

これらの操作の多くはクロードコードが代わりに行ってくれます。あなたが覚えるべきは「保存して」「公開して」と日本語でお願いすればよい、ということです。

### 6.6 困ったときのチェックリスト

- [ ] 画面に何も表示されない → スプレッドシートの共有設定が「閲覧者」になっているか確認する
- [ ] 「アクセスが拒否されました」と出る → 会社のGoogleアカウントでログインした状態でシートを開き直し、共有範囲を確認する
- [ ] 一部の列だけ反映されない → スプレッドシートの列名(見出し)が、クロードコードに伝えた名前と一致しているか確認する
- [ ] スマホだけレイアウトが崩れる → その旨をスクリーンショット付きでクロードコードに伝える
- [ ] それでも解決しない → エラーメッセージ全文をコピーして先輩に共有する

- 個人情報や社外秘データの扱いに迷ったとき → 必ず先輩・管理者に確認してから進める

### 7.4 用語集

| 用語 | 説明 |
|---|---|
| プロンプト | クロードコードへのお願い文・指示文のこと |
| MVP | Minimum Viable Product の略。最初に作る「最低限動くもの」 |
| レスポンシブ対応 | パソコンでもスマホでも見やすいように画面が自動で調整されること |
| リポジトリ | アプリのファイル一式を入れておく専用フォルダ |
| コミット / プッシュ | 変更の保存 / 保存した変更をリポジトリに反映すること |
| GitHub Pages | 作ったアプリをインターネット上のURLとして公開する仕組み |
| Google Apps Script(GAS) | Googleスプレッドシートなどを操作するための仕組み。書き込み機能を作るときに使う |

### 7.5 プロンプトテンプレート集(コピペ用)

**新しいアプリを頼むとき:**
```
【背景】
【データ】
【欲しいもの】
【使う場面】
【最初の一歩】
```

**機能を追加してほしいとき:**
```
今のアプリに、次の機能を追加してください。
・追加してほしいこと:
・理由(背景):
・完成イメージ:
```

**不具合を直してほしいとき:**
```
次の不具合を直してください。
・起きていること:
・本来こうなってほしい:
・(あればエラーメッセージ全文をここに貼る)
```

**スマホ表示を直してほしいとき:**
```
スマホで見たときに次の点が気になります。直してください。
・気になる箇所:
・添付のスクリーンショットの状態:
・理想の見え方:
```
